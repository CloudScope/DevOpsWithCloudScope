Q1. Walk me through designing a multi-tenant EKS cluster for production —
    namespaces, RBAC, network policies, and resource isolation.
    [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  I map each tenant to a namespace and enforce isolation at network, RBAC, and
  resource layers — nothing shared by default.

ANSWER:
  Each tenant gets a dedicated namespace. RBAC roles are scoped per namespace
  using RoleBindings — no cluster-admin is handed out. NetworkPolicies
  default-deny all ingress and egress, with explicit allow rules only for
  approved cross-namespace traffic. ResourceQuotas and LimitRanges prevent
  noisy-neighbour issues. A single ALB Ingress Controller at cluster level
  handles ingress with host/path-based routing and TLS termination. Pod
  Security Admission policies block privileged containers and root users. Node
  groups are segmented by workload type — Spot for stateless, On-Demand for
  stateful.

Q2. Explain Kubernetes resource requests vs limits. What happens when a node
    runs out of memory and how do you tune these in production?
    [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Requests are what the scheduler uses to place a pod; limits are the ceiling
  enforced by the kubelet at runtime.

ANSWER:
  Requests reserve capacity on the node — the scheduler only places a pod if
  enough is available. Limits are enforced at runtime: CPU is throttled when
  exceeded, but memory OOM-kills the container when it breaches the limit. In
  production I use VPA in recommendation mode to collect usage data, then set
  requests at p95 actual usage and limits at 1.5–2x requests. For memory I
  set limits tighter because OOM kills are hard to debug — I alert on
  container_memory_working_set_bytes approaching the limit. Pods without
  requests go into BestEffort QoS and get evicted first under pressure — I
  never leave requests unset in production.

YOUR ANGLE:
  Tie this to your Grafana monitoring work — explain how you alert on resource
  saturation before it causes incidents.

Q3. How does Kubernetes handle rolling updates and how do you configure
    truly zero-downtime deployments?
    [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Rolling updates replace pods gradually, but zero-downtime requires readiness
  probes, preStop hooks, and careful traffic draining.

ANSWER:
  I set strategy type RollingUpdate with maxSurge 25% and maxUnavailable 0 for
  critical services so capacity never drops. ReadinessProbes ensure traffic
  only routes to healthy pods. preStop hooks add a sleep of 5–15 seconds so
  in-flight requests drain before the pod terminates. minReadySeconds ensures
  the new pod is stable before proceeding. For schema changes I use
  backward-compatible migrations first — deploy the migration job, then the
  app, then remove old columns. I also set terminationGracePeriodSeconds higher
  than the longest expected request.

  Tie this to your Spinnaker canary experience at JP Morgan and GitHub Actions
  pipelines at Advance Auto Parts.

Q4. How do you approach EKS cluster version upgrades with zero downtime
    across a production environment?
    [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Cluster upgrades are a multi-phase operation — control plane first, then node
  groups — with API deprecation checks before anything moves.

ANSWER:
  I run kubent first to detect deprecated API usage in existing manifests and
  fix them before upgrade. I upgrade the EKS control plane first — AWS manages
  the API server but I validate add-ons compatibility. Then I upgrade node
  groups one at a time using blue-green node groups: spin up new nodes at the
  target version, cordon old ones, and let the Cluster Autoscaler drain
  workloads over. PodDisruptionBudgets on all critical workloads prevent too
  many pods going down during drain. I test on staging one minor version ahead
  and check all Helm chart compatibility. Add-ons like CoreDNS, kube-proxy,
  and VPC CNI are upgraded in sequence after the control plane.

YOUR ANGLE:
  Your CKA certification is directly relevant here — cluster lifecycle
  management is a core CKA domain.

Q5. How do you implement Horizontal Pod Autoscaling beyond CPU — what custom
    metrics have you used?
    [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  CPU-based HPA is a blunt instrument — I prefer scaling on business metrics
  that actually reflect load, using KEDA or custom metrics adapters.

ANSWER:
  I use KEDA (Kubernetes Event-Driven Autoscaling) to scale on metrics from
  external sources — SQS queue depth, Kafka consumer lag, or custom Prometheus
  metrics. For a payment processing service I scaled on pending transactions
  per second from a Prometheus metric rather than CPU, because CPU lagged
  behind actual load by 30–60 seconds. KEDA scales to zero for event-driven
  workloads, which saves significant cost. For the Prometheus adapter I define
  custom metric rules in a ConfigMap and expose them via the metrics API. I
  always pair HPA with PodDisruptionBudgets to prevent scale-down from removing
  too many pods at once.

Q6. Explain how Kubernetes networking works — from a pod making a request to
    an external service. What are CNI plugins and how do you choose one?
    [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Kubernetes networking is built on the CNI spec — every pod gets a routable
  IP and the CNI plugin is responsible for making that work.

ANSWER:
  When a pod is scheduled, the kubelet calls the CNI plugin to allocate an IP
  and configure the network interface. On AWS EKS I use the VPC CNI plugin
  which assigns actual VPC IPs to pods — this means pods are first-class
  network citizens, routing to RDS or other AWS services works natively without
  NAT, and security groups can be applied at the pod level. Traffic from pod to
  external service goes: pod → node → VPC routing → target. For CNI choice:
  VPC CNI is best on EKS for AWS service integration; Calico adds network
  policy enforcement; Cilium uses eBPF for higher performance and deep
  observability. I choose based on scale, observability needs, and whether I
  need L7 policy enforcement.

  Mention your VPC and subnet architecture design experience at JP Morgan —
  this shows depth beyond just deploying workloads.

Q7. How do you handle stateful workloads in Kubernetes — databases or message
    brokers? What are the trade-offs vs managed services?
    [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Running stateful workloads in Kubernetes is possible with StatefulSets and
  persistent volumes, but for databases I almost always prefer managed services.

ANSWER:
  For stateful apps I use StatefulSets which give stable network identities,
  ordered pod startup/shutdown, and persistent volume claims that survive pod
  restarts. I use StorageClasses backed by EBS gp3 volumes with appropriate
  IOPS and throughput settings. However, for production databases I strongly
  prefer managed services — RDS PostgreSQL for relational, ElastiCache for
  Redis — because they handle replication, failover, backups, and patching out
  of the box. The operational overhead of running a highly available database
  in Kubernetes (replication lag monitoring, failover testing, backup
  validation) is rarely worth it unless you have very specific requirements. I
  have administered PostgreSQL on AWS RDS at Advance Auto Parts which gives me
  a concrete basis for this preference.

YOUR ANGLE:
  Your PostgreSQL RDS admin experience is a strong supporting point here —
  use it to show you have operated stateful workloads at scale.

Q8. How do you structure Terraform for a large organisation with multiple
    teams, environments, and shared services?
    [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  The key is separating state boundaries by blast radius, enforcing module
  reuse, and giving teams autonomy within guardrails.

ANSWER:
  I structure Terraform in layers: a foundation layer for VPC, IAM, and shared
  networking with its own state file; a platform layer for EKS, RDS, and
  shared services; and team-level layers for application workloads. Each layer
  has its own remote state in S3 with DynamoDB locking. Teams consume shared
  infrastructure via remote state data sources — never direct resource
  references across layers. Reusable modules live in a versioned private
  registry; teams pin to specific semver versions. CI/CD enforces terraform
  fmt, validate, and plan on PRs with the plan output posted as a comment.
  Apply only runs from main after approval. I use Sentinel or OPA for policy
  as code — blocking public S3 buckets, enforcing tagging, and requiring
  encryption.

Q9. What is GitOps and how does it differ from traditional CI/CD? How would
    you implement it for a Kubernetes platform?
    [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  GitOps treats Git as the single source of truth for cluster state, with an
  agent continuously reconciling actual state toward declared state.

ANSWER:
  Traditional CI/CD is push-based — a pipeline pushes changes to an
  environment. GitOps is pull-based — an agent inside the cluster (ArgoCD or
  Flux) watches a Git repo and reconciles any drift. Benefits: no outbound
  credentials from CI, full audit trail in Git history, rollback is a git
  revert. My implementation: application teams write Helm charts committed to
  a config repo. Environment-specific values in separate directories using
  Kustomize overlays. GitHub Actions pipeline runs tests, builds the image, and
  bumps the image tag in the config repo via a PR. ArgoCD detects the change
  and syncs the cluster. I configure ArgoCD with auto-sync and self-heal so
  any manual cluster change is reverted automatically.

YOUR ANGLE:
  This role specifically lists GitOps — be explicit about the pull-based model
  and how your GitHub Actions experience feeds into it.

Q10. How do you manage secrets securely in a GitOps or IaC workflow?
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Secrets should never live in Git, even encrypted — the goal is to store only
  the reference in Git and the actual secret outside.

ANSWER:
  I use the External Secrets Operator to sync secrets from AWS Secrets Manager
  into Kubernetes Secrets at deploy time. The GitOps repo only contains the
  ExternalSecret manifest which defines what to fetch and where to put it —
  never the value. For Terraform, sensitive variables come from environment
  variables or AWS Parameter Store, sensitive outputs are marked sensitive, and
  state is encrypted at rest in S3 with a KMS key. For CI/CD I use GitHub
  Actions OIDC federation with an AWS IAM role — no long-lived access keys
  stored anywhere. I audit secret access via CloudTrail and rotate secrets
  automatically using Secrets Manager rotation lambdas.

  Connect to your DevSecOps experience and GitHub Advanced Security familiarity
  — secret scanning in pipelines is part of your story.

Q11. How do you detect and remediate infrastructure drift in a
     Terraform-managed environment?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Drift happens when someone changes infrastructure outside Terraform — the
  solution is continuous detection and a culture that closes the loop.

ANSWER:
  I run terraform plan on a schedule — daily via a CI cron job — and alert via
  Slack or PagerDuty if the plan shows unexpected changes. AWS Config rules
  detect out-of-band console changes and publish findings to SNS. To prevent
  drift I use SCPs or IAM permission boundaries to restrict direct console
  write access in production — all changes must go through Terraform via PR.
  If drift is detected I create a ticket, review the change intent, and either
  import it into Terraform state or raise a PR to revert it. I use Infracost
  in the pipeline to flag cost drift alongside infrastructure drift.

Q12. How do you implement policy as code for cloud infrastructure?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Policy as code shifts compliance left — catching violations at plan time
  rather than after resources are created.

ANSWER:
  I use OPA (Open Policy Agent) with Conftest to evaluate Terraform plans and
  Kubernetes manifests against policy rules written in Rego. Policies cover:
  all S3 buckets must have encryption and block public access, all EC2
  instances must have required tags, no security groups allow 0.0.0.0/0 on
  port 22, all Kubernetes deployments must have resource requests set. These
  run as a required check in the GitHub Actions pipeline — the plan fails if

  Gatekeeper with ConstraintTemplates that enforce the same policies at
  admission time as a second line of defence.

Q13. Compare Terraform and CloudFormation. When would you choose one over
     the other?
     [Difficulty: Foundational]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Both are mature IaC tools but they have different philosophies — Terraform is
  multi-cloud and modular, CloudFormation is AWS-native and tightly integrated.

ANSWER:
  I choose Terraform when: the organisation uses multiple cloud providers, the
  team wants a rich module ecosystem and provider community, or we need
  fine-grained state management and workspace support. I choose CloudFormation
  when: the workload is purely AWS and we need native integration with AWS
  Service Catalog, StackSets for multi-account deployments, or when the team
  is already invested in CDK. Terraform's HCL is more readable and its
  plan/apply cycle is cleaner than CloudFormation's change sets. However
  CloudFormation has no state file to manage and has native drift detection.
  In practice I have used Terraform at scale and find it more flexible for
  platform teams managing complex multi-service infrastructure.

YOUR ANGLE:
  You have experience with both — be honest about trade-offs rather than
  saying one is always better.

Q14. How do you define and implement SLOs, SLIs, and error budgets for a
     microservices platform? Give a concrete example.
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  SLIs are what you measure, SLOs are the targets you commit to, and error
  budgets are what you spend when you take risk.

ANSWER:
  For a payment API I define SLIs as: availability (successful 2xx responses /
  total requests), latency p99, and error rate. The SLO: 99.9% availability,
  p99 latency under 500ms, error rate under 0.1%. I instrument these using
  Prometheus — histogram_quantile for latency, recording rules for availability
  ratio. Grafana dashboards show current SLO status and burn rate.
  Multi-window burn rate alerts: fast burn (2% budget in 1 hour) triggers
  PagerDuty immediately; slow burn (5% budget in 6 hours) triggers a warning.
  The error budget gates releases — if budget is exhausted the team freezes
  non-critical deployments until it recovers. This makes reliability a shared
  team metric, not just an ops concern.

Q15. Walk me through how you conduct a blameless Root Cause Analysis after
     a P1 production incident.
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  A good RCA is blameless, structured, time-boxed, and always results in
  systemic fixes — not individual blame.

ANSWER:
  During the incident I keep a running timeline in a shared document —
  timestamps, actions taken, who did what. Within 24 hours of resolution I
  send a brief incident summary to stakeholders. Within 48 hours I schedule a
  postmortem with all involved parties. The postmortem covers: incident
  summary, timeline of events, root cause using 5-whys, contributing factors,
  what went well, what did not, and action items. Action items must be SMART —
  assigned to a named owner with a due date and a success metric. I track them
  in Jira and review closure at the next postmortem. I look for patterns across
  incidents — if the same component causes multiple incidents it escalates to a
  dedicated reliability sprint.

  identified and the preventive measure you implemented.

Q16. How would you implement chaos engineering in a production environment?
     How do you get stakeholder buy-in?
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Chaos engineering is controlled experimentation to build confidence in
  resilience — not random destruction.

ANSWER:
  I use Gremlin which I have deployed at JP Morgan. The process: define steady
  state using baseline metrics, form a hypothesis (killing one pod in a
  3-replica deployment should cause zero user impact), execute the experiment
  with a defined blast radius starting in staging, observe against the
  hypothesis, fix gaps found, then graduate to production. Each experiment has
  a runbook and a kill switch. For stakeholder buy-in I frame it as: we are
  choosing to find failures before our customers do, during business hours,
  with full control, rather than discovering them at 2am with no context. I
  share data on MTTR reduction after each experiment — chaos engineering has
  directly improved our recovery time because we practised recovery.

Q17. How do you approach capacity planning for a Kubernetes platform —
     both compute and cost?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Capacity planning is about staying ahead of demand without over-provisioning
  — data-driven, not gut-feel.

ANSWER:
  I collect 90-day trends from Prometheus for CPU, memory, and business metrics
  like requests-per-second. I project forward using growth rate from the last
  quarter and plan for 2x peak headroom. Cluster Autoscaler handles reactive
  scaling of node groups; I tune scale-down delay to prevent thrashing. HPA
  scales pods based on custom metrics via KEDA. For cost I use a mix of
  On-Demand for baseline and Spot for burst — Spot interruption handling via
  node termination handler ensures graceful drain. I run quarterly load tests
  at 150% projected peak to validate headroom. AWS Cost Explorer tracks spend
  by tag (team, environment, service) and I review with team leads monthly
  using anomaly detection alerts.

Q18. What is your approach to on-call rotation design and reducing on-call

     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:

ANSWER:

Q19. How do you ensure high availability and disaster recovery for a platform
     running across AWS regions?
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  High availability requires active-active or active-passive design with
  automated failover — disaster recovery adds a tested recovery playbook.

ANSWER:
  For HA I design active-active across two AWS regions with Route53
  latency-based or health-check failover routing. EKS clusters in each region
  are independent — no cross-region kubelet communication. Data replication uses
  RDS Multi-AZ within region and cross-region read replicas for DR. For
  stateless services the second region can take traffic within 30 seconds via
  Route53 health check failover. For stateful services I define RPO and RTO
  targets with the business — typically RPO 1 hour, RTO 4 hours — and design
  backup and restore to meet them. I run quarterly DR tests where we simulate
  a full region failure, promote the secondary, validate functionality, and
  document the actual RTO achieved. The runbook is versioned in Git.

YOUR ANGLE:
  Connect to your AWS VPC architecture and multi-region design experience at
  JP Morgan — this is exactly the kind of infrastructure you have built.

Q20. Design a full CI/CD pipeline for a microservice — from commit to
     production — with security scanning, testing, and canary deployment.
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  I design this as a multi-stage GitHub Actions pipeline with quality gates at
  every stage and automated rollback on anomaly detection.

ANSWER:
  Stage 1 — PR checks: unit tests, SonarQube SAST scan, Trivy container image
  vulnerability scan, Conftest policy checks on Kubernetes manifests. PR
  blocked if any critical finding.
  Stage 2 — Merge to main: build and push image to ECR with commit SHA tag,
  deploy to staging via Helm with ArgoCD, run integration and smoke tests, run
  DAST scan against staging.
  Stage 3 — Canary: deploy to 5% of production traffic via weighted ingress or
  Argo Rollouts. Monitor error rate, p99 latency, and business metrics for 15
  minutes via Prometheus. If burn rate exceeds threshold, automated rollback
  triggers via ArgoCD.
  Stage 4 — Full rollout: promote to 100% if canary is healthy. All stages
  emit deployment markers to Grafana for metric correlation.

YOUR ANGLE:
  This maps directly to your Spinnaker canary work at JP Morgan and GitHub
  Actions pipelines at Advance Auto Parts — use both as examples.

Q24. How do you implement progressive delivery — blue-green, canary, and
     feature flags?
     [Difficulty: Advanced]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Progressive delivery reduces deployment risk by controlling the blast radius
  of a bad release — I use different strategies depending on risk profile.

ANSWER:
  Blue-green: I maintain two identical environments, switch traffic at the load
  balancer or Route53 level. Good for infrastructure changes where I want
  instant cutover and instant rollback. Cost is doubled capacity during the
  switch. Canary: I route a small percentage of traffic to the new version
  using weighted ingress rules or Argo Rollouts, gradually increasing based on
  metric health. Good for application changes where I want to validate with
  real traffic. Feature flags: I use LaunchDarkly or a config-driven approach
  to decouple deployment from release — I can deploy code to all users but
  activate the feature only for a subset. This is especially useful for
  high-risk features. I have used Spinnaker at JP Morgan for canary analysis
  with automated promotion and rollback based on Kayenta metric analysis.

  production is visible and reviewable in Git. I use external configuration via
  ConfigMaps and Secrets sourced from AWS Parameter Store, not baked into
  images. Config changes follow the same PR process as code changes — reviewed

Q26. How do you build an observability platform from scratch for a
     microservices architecture?
     [Difficulty: Advanced]

OPENING LINE:

Q27. How do you reduce alert fatigue in a production monitoring environment?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Alert fatigue happens when teams stop trusting alerts — the fix is
  symptom-based alerting tied directly to SLOs.

ANSWER:
  I shift from cause-based alerts (CPU > 80%) to symptom-based alerts (error
  rate > 1% sustained for 5 minutes). I use multi-window multi-burn-rate
  alerting: fast burn for critical pages, slow burn for warnings. I audit all
  alerts quarterly — any alert that has fired but required no action in 90 days
  gets deleted or converted to a dashboard metric. Alertmanager inhibition rules
  suppress downstream alerts when the root cause alert fires — if a node is
  down, suppress all pod alerts on that node. Every alert links to a runbook.
  I track the page-to-action ratio: if a page fires but on-call takes no
  meaningful action, the alert is broken. My target is zero alerts that result
  in no action.

YOUR ANGLE:
  Reference your MTTD reduction work at JP Morgan and the Grafana/Splunk
  alerting frameworks you built.

Q28. Explain distributed tracing — how does it work and how do you implement
     it in a microservices platform?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Distributed tracing follows a request as it flows across multiple services,
  recording timing and metadata at each hop.

ANSWER:
  When a request enters the system a trace ID is generated and propagated in
  request headers (W3C TraceContext standard). Each service creates a span for
  its processing, recording start time, end time, service name, and metadata,
  then sends it to a trace collector. The collector assembles spans into a
  trace showing the full request path and where time was spent. I implement
  this using the OpenTelemetry SDK — it provides auto-instrumentation for
  common frameworks (HTTP clients, gRPC, databases) so most tracing is
  zero-code. Traces go to Jaeger or AWS X-Ray. I sample at 100% for error
  traces, 10% for slow traces, and 1% for normal traces to manage storage
  cost. Trace IDs are injected into logs so I can jump from a log line to the
  full trace.

YOUR ANGLE:
  Show you understand the implementation detail — auto-instrumentation vs
  manual spans, and the sampling strategy decisions.

Q29. How do you use Splunk effectively for production troubleshooting? Walk
     me through investigating a latency spike.
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Splunk is most powerful when logs are structured — I can then use SPL to
  correlate, aggregate, and pivot across services in seconds.

ANSWER:
  When a latency spike alert fires I start in Grafana — I identify which
  service's p99 latency spiked and the exact time window. I switch to Splunk

  search that service's logs for slow responses. I use Splunk's transaction
  command to group log lines by trace ID and measure request duration
  end-to-end. I save the search as a dashboard panel so the next person
  investigating a similar issue has a starting point. I annotate the Grafana
  dashboard with the incident marker for future correlation.

Q31. How do you implement log aggregation at scale across a large Kubernetes
     cluster?
     [Difficulty: Intermediate]
────────────────────────────────────────────────────────────────────────────────

OPENING LINE:
  Log aggregation at scale requires a reliable collection agent, structured log
  format, and a pipeline that handles spikes without data loss.

ANSWER:
  I deploy Fluent Bit as a DaemonSet — it is lightweight and handles
  back-pressure better than Fluentd at high volume. Fluent Bit reads container
  logs from the node's /var/log/containers, enriches them with Kubernetes
  metadata (namespace, pod name, container name, node name) using the
  Kubernetes filter plugin, and ships to Splunk HEC or OpenSearch. I enforce
  structured JSON logging via a shared logging library so every log has a
  consistent schema: timestamp, level, service, trace_id, message, and context
  fields. I set resource limits on Fluent Bit pods to prevent log collection
  from impacting application workloads. For high-throughput services I use
  Kafka as a buffer between Fluent Bit and Splunk to handle ingestion spikes.
  Retention policies are set per index based on compliance requirements.

YOUR ANGLE:
  You have used both Splunk and OpenSearch — mention both and show you
  understand the trade-offs between them.
