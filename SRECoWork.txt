Energy: TECHNICAL CONFIDENCE. Show you built it.
================================================================================

"Here's what we actually built — and I want to be clear about the real
technical choices we made.

Left side: three purpose-built agents. Each one doesn't just dump raw data —
it pre-processes. The Splunk agent classifies severity and attaches deployment
context. The OTel agent maps service topology so the LLM understands blast
radius. Dynatrace gives infra metrics with Davis AI baselines already applied —
so we're not reacting to raw numbers, we're reacting to deviations from
expected behavior.

Center: the LLM engine. This is where we spent the most time. Prompt
engineering for multi-signal correlation is genuinely hard. Getting the LLM
to reason across log patterns, trace anomalies, and infra metrics
simultaneously — and produce a ranked, actionable RCA — took real iteration.

Bottom: the knowledge base. Confluence, Stack Overflow, engineering wiki,
public tech blogs. The LLM doesn't just retrieve — it reasons over what it
retrieves and decides what's relevant to this specific incident.

Right side: two outputs. Kubernetes MCP for action. Confluence for
documentation.

The architecture decision we're most proud of? Using MCP instead of direct
kubectl access. It gives us a structured, policy-enforced, auditable action
layer. The LLM can't just do anything — it can only do what the MCP allows.
That's the guardrail that makes this safe to run autonomously in production."

SLIDE 5 — OBSERVABILITY INGESTION  (~2 minutes)
Energy: TECHNICAL BUT GROUNDED. Show the depth.
================================================================================

"Let me quickly tell you why we chose these three sources and not just one.

Most teams pick one observability tool and go deep. The problem is each tool
sees a different layer of truth.

Splunk sees what the application said happened.
OTel sees how the failure propagated across services.
Dynatrace sees what the infrastructure was doing when it happened.

A memory error in Splunk without the Dynatrace infra view — you don't know
if it's a leak or a traffic spike. A latency spike in OTel without the Splunk
errors — you don't know if it's a downstream timeout or an upstream retry
storm. You need all three to reason correctly.

That's why we built three separate agents — each optimized for its data
source — feeding a unified context window to the LLM.

During the hackathon we actually hit a real challenge here: context window
management. You can't dump all your logs and traces into a single LLM prompt.
We had to build a pre-processing layer that summarizes, deduplicates, and
selects the most signal-rich data to forward. That's a real engineering
problem. We solved it."


================================================================================
SLIDE 6 — LLM-POWERED ANALYSIS  (~3 minutes)
Energy: THIS IS YOUR TECHNICAL CENTREPIECE. Slow down here.
================================================================================

"This is the brain. And I want to be honest — this is where we spent 60% of
our hackathon time, because this is genuinely hard to get right.

The five-step analysis pipeline:

SIGNAL FUSION — correlate events across three data sources into one incident
timeline. Deduplicate. If 10 pods log the same error, that's one incident
pattern, not ten noisy signals.

ANOMALY SCORING — severity, blast radius, is this growing or stabilizing?
We tuned this to avoid the trap of the LLM treating every anomaly as a P1.

ROOT CAUSE ANALYSIS — this is the hard one. We're asking the LLM to reason
causally across application, network, and infrastructure layers. We prompt it
to produce a causal chain, not just a symptom list.

KNOWLEDGE RETRIEVAL — the LLM queries Confluence for runbooks, Stack Overflow
for similar error threads, engineering wikis for ownership info, and can hit
public engineering blogs for novel issues. This is RAG done properly — not
just semantic similarity, but relevance to this incident's specific signature.

REMEDIATION RECOMMENDATION — ranked action plan, with confidence scores,
immediate automated steps, and manual fallback.

[Point to the RCA output on slide]

This is real output from our system. Payment-service OOMKilled. Root cause
identified — memory limit exceeded during a traffic spike, compounded by a
memory leak in v2.3.1. Actions recommended and executed. MTTD: 47 seconds.

The confidence score is real — we calibrated it against historical incidents.
94% means the LLM correctly identified root cause and the right fix 94% of
the time on our test dataset."

SLIDE 7 — KUBERNETES MCP REMEDIATION  (~3 minutes)
Energy: THIS IS YOUR INNOVATION DIFFERENTIATOR. Own it.
================================================================================

"I want to spend a bit of time here because this is what makes our solution
different from just another monitoring dashboard.

A lot of AIOps tools detect problems. Some even diagnose them. Very few
act — and when they do, it's usually via rigid, pre-scripted automation that
breaks the moment the incident is slightly outside the expected pattern.

We used Kubernetes MCP — the Model Context Protocol — to give the LLM
structured, safe, context-aware access to the cluster.

Here's why MCP and not direct kubectl: because we needed a layer of intent
validation between the LLM's decision and the cluster's reality. The MCP
defines exactly which operations are available, validates parameters, enforces
policy guardrails, and returns structured results the LLM can reason about.

So the LLM can't delete a StatefulSet in production. It can't scale beyond
pre-defined limits without a human approval flag. It can't touch database
pods. The guardrails are built into the MCP schema — not into a prompt that
could be reasoned around.

What it CAN do: delete crashing pods, increase resource quotas, modify
deployments, adjust HPA thresholds.

During the hackathon we had a real debate about how much autonomy to give
the LLM. Our answer: full autonomy for reversible, well-understood remediation
patterns. Human-in-the-loop flag for anything destructive or novel. That's
the right balance for a system you'd actually trust in production."

Why does this matter for a hackathon? Because compounding value. Every

JUDGE Q&A — ANTICIPATED QUESTIONS & ANSWERS
================================================================================

Q: How do you handle false positives?
A: "The LLM scores confidence. Below a threshold, it flags for human review
   instead of acting. The guardrails in the MCP mean no irreversible action
   happens without high confidence."

Q: What if the LLM gets the RCA wrong?
A: "The audit trail captures every decision and signal that led to it.
   Engineers can review and correct. Each correction feeds back as a
   training signal."

Q: How does this scale across 200 services?
A: "The agents are stateless and horizontally scalable. The LLM context is
   bounded per incident, not per service. Scaling is an infra problem,
   not an architecture problem."

Q: What is the latency of the LLM call?
A: "Under 30 seconds for analysis. We batch the signals, not the LLM calls.
   One call per incident, not per alert."

Q: Why MCP and not just direct API calls?
A: "Policy enforcement, structured schema, and auditability. MCP is the
   right abstraction between LLM intent and cluster action."

Q: Is this production-ready or just a hackathon demo?
A: "The architecture is production-ready. The agents, MCP integration, and
   LLM pipeline are built on real tooling. What we'd add before shipping:
   more extensive guardrail policies and a broader test suite."

Q: What LLM are you using?
A: "We used [your model here — Claude / GPT-4 / etc.]. The system is
   model-agnostic by design — the prompt layer abstracts the LLM so we
   can swap models without changing the pipeline."
